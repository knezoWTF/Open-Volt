## 📖 About Volt

OpenVolt is a **free and open-source** Minecraft client built for version 1.21.1, specifically designed and optimized for 1.9+ combat mechanics. Originally conceived as a premium client, Volt was made open-source to empower the Minecraft community to collaborate and create the best possible client experience.

### Why OpenVolt?

- **🎯 Combat-Focused**: Tailored for 1.9+ combat mechanics with precise, legitimate features
- **🔒 Bypass-Oriented**: Features designed to be undetectable on major servers
- **🌐 Community-Driven**: Built by the community, for the community
- **🚫 No Blatant Features**: Focused on legitimate gameplay enhancement, not cheating
- **📜 Open Source**: Transparent development under GPL v3 license

## 🚀 Installation

### Prerequisites
- Java 21 or higher
- Minecraft 1.21.1
- Fabric Loader

### Setup the project

1. **Clone the repository `git clone <url>`**
2. **Run `gradle genSources` to setup the project**
3. **Compile using `gradle build`**

## 🤝 Contributing

We welcome contributions from the community! However, to maintain code quality and project direction, please follow these guidelines carefully.

### 🎯 Contribution Guidelines

#### ✅ What We Accept
- Legitimate, bypass-focused features
- Performance optimizations
- Bug fixes and stability improvements
- Documentation improvements
- GUI enhancements (must be optional, not default)
- Code quality improvements (within reason)

#### ❌ What We Don't Accept
- **Blatant features** (e.g., killaura, fly, speed hacks)
- **Useless features** that don't add value
- **Mass refactoring** or restructuring of packages
- **Folder structure changes** without prior approval
- Features that compromise server bypass capability

### 📝 Commit Message Guidelines

Use clear, descriptive commit messages following this format:

```
<type>: <short description>

<optional detailed description>
```

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `perf`: Performance improvement
- `docs`: Documentation changes
- `style`: Code style changes (formatting, etc.)
- `refactor`: Code refactoring (small-scale only)
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

**Examples:**
```
feat: add velocity module with bypass mode
fix: resolve click simulation timing issue
perf: optimize rendering performance by 15%
docs: update installation instructions
```

### 🌿 Branch Naming Convention

Create branches using this naming pattern:

```
<type>/<short-description>
```

**Examples:**
```
feat/velocity-module
fix/click-timing
perf/render-optimization
docs/readme-update
```

### 🔄 Pull Request Process

1. **Fork** the repository
2. **Create a branch** following the naming convention above
3. **Test thoroughly** - Ensure your feature works and bypasses anti-cheat
4. **Write clear descriptions** - Explain what your PR does and why
5. **Submit PR** - Wait for review and be responsive to feedback
